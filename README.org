    |-- nix              # Nix tools
    |-- programs         # Discrete, self-contained features. Like Niri, or Noctalia.
    |-- system           # Building generic desktops and settings, which are services needed by the desktops
    |-- users            # Yup, you guessed right!
#+end_src

* Dendriwhat now?

My dotfiles are currently modeled after the dendritic pattern, which is a way to organize your system from the bottom up instead of top down, designing discrete "features" (it just means standalone) that are then collected either hierarchically (into system types) or picked up by import-tree. That gives me a very flexible, scalable config that's very neat to play with.
Basically, we're using [[https://github.com/hercules-ci/flake-parts][Flake-parts]] to separate our config in discrete "modules" that can then be imported automatically by [[https://github.com/vic/import-tree][import-tree]]. That leaves us with a very clean flake and a clean, well-separated config. It also allows, if done right, to scale easily with each new system you bring in the config.

It's definitely the buzzword at the moment, but it's also very cool, so hype wagon it is.

* The goal: A literate config. I am not kidding

In the coming days, I'll be switching as much as possible my config to org files (emacs) that will tangle and create the nixos configuration. So...why?
What's fun about org files is that I can create beautiful, legible documents that will serve me in two ways.
+ Great documentation is a boon for me and the two or three people that will also see this.
  Sometimes, comments in code don't give me enough freedom to write, as it obscures the code in a sea of uncollapsible comments.
  Org-mode changes that. It's a bit like markdown, so I can make parts of a file collapsible, with headers, make clickable links to relevant information, etc..
+ Force me to actually document everything I should be documenting so that if I come back to a file a year later, I still know what everything does.
