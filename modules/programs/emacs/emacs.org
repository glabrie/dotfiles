I'm mainly a hobbyist. I like diving into new things and learning.
Emacs is a program released in 1985, {{{results(emacs-age)}}} years ago. It's a bit older than me, by some months. Being from another time but still actively developed and maintained to this day, Emacs evolved into its own beast, the product of a divergent evolution with its very own ethos. Growing in isolation from UI/UX trends and standards over these four decades, Emacs forged its own path. A very different architecture creates a nomenclature so alien it sometimes feels obscure, almost occult in a way. There is definitely something magical in there.

In {{{results(emacs-age)}}} years, I've changed so much that it might be hard to chart me as a single individual line. Not so with Emacs. Its roots are still there and vibrant.

Emacs sells itself as a text editor, but I would argue against that. It's really an Elisp interpreter that went haywire trying to recreate an OS. That interpreter affords us the magic that Emacs summons as it becomes this mutable platform that blurs the line between writing in an application and writing the application itself.

Now, that Lisp machine can do anything your mind conjures. Years of hacks, packages, extensions are at your fingertips. Get lost in a new world.

** Opacity problems
:PROPERTIES:
:CUSTOM_ID: issue-opacity
:END:

- Last investigated :: [2026-05-04]
- Severity :: Cosmetic

*** Description of the issue

You can now apply transparency to emacs's background natively with this bit:

#+begin_src emacs-lisp
(set-frame-parameter nil 'alpha-background 70)
(add-to-list 'default-frame-alist '(alpha-background . 70))
#+end_src

Note that you need a PGTK package of emacs for it to work on Wayland. With that done, you should have a nice transparency effect on the background only of emacs' frames. The issue I am having with it is in regard to the leading asterisks behind the header levels.
Normally, headers are denoted with stars. The depth is denoted by the number of stars, like so:

#+begin_example
,* Level 1
,** Level 2
,*** Level 3
#+end_example

Org-mode will normally attempt to hide the stars by using the ~Org-hide~ [[*Faces][face]], and then use ~org-indent-mode~ to indent them visually.
Concretely, when ~org-hide-leading-stars~ is set to ~t~, a font-lock keyword is installed that will apply ~org-hide~ to the stars. That face has a ~:foreground~ color property that ~Org~ will set to match the frame's ~:background~ color property, making them appear invisible. This would normally be fine, except when transparency is turned on. Now that background is not static, and your background has an opacity value, the hidden stars shine through at 100% opacity, leaving faint black marks behind the bullet that you want shown, like here:

[[../../../assets/issue-opacity-1.png]]

*** Attempted fixes and knowledge gained.

+ I initially tried to play with ~org-modern-hide-stars~, setting it to ~nil~ or to or to ~" "~.
+ Disabled ~org-startup-indented~ to diagnose. It did remove the problem at the cost of visual indenting.

**** Incidental knowledge

+ ~org-modern-hide-stars " "~ is a clean alternative that bypasses the face mechanism entirely but for some reason seem to disable itself when ~org-indent-mode~ is active, as far as I could see.
+ ~org-indent-mode~ which is enabled by ~org-startup-indented t~, seems to reset any global changes made by ~org-modern-hide-stars t~ at the local buffer level, preventing it from firing at all.

*** Resolution

As of [2024-05-04] I chose to remove transparency from emacs. I will revisit this someday.

*** Emacs as a server

One of the understated advantages of emacs is being able to run it as a server and have new frames attach to it automatically. This has several benefits here:
- Doom Emacs is started just the once, meaning that actually opening an emacs frame (the app) is, wait for it, *Blazingly Fast*.
- Since the daemon is running in the background, processes like mu4e or the killring (copy-paste) don't have to restart between frames, you just reattach each time. It also means you can have several frames opened that link to the same buffers (Let's say you have one instance and another in the terminal for example).
- I also use it as a global org-mode capture with a hotkey in niri ( ~Mod+C~ ) that make noting things on the fly a breeeze.

We'll also start mbsync, which is necessary for mu4e.

#+begin_src nix
    services = {
      emacs.enable = true;
      mbsync.enable = true;
    };
#+end_src

***** Dynamic splash sizing

Scale the splash image to 40% of the frame's pixel width when each frame is created. ~after-make-frame-functions~ fires before the dashboard renders, so the resized path is picked up on first display. Works correctly with the Emacs daemon since the frame has its real pixel width by the time this runs.

#+begin_src emacs-lisp
(defun +my/scale-splash-image (&optional _frame)
  (let* ((width (round (* (or (ignore-errors (x-display-pixel-width)) 1920) 0.35)))
         (src (expand-file-name "~/.dotfiles/assets/aperture40s.png"))
         (dest "/tmp/doom-aperture-splash.png")
         (bg (let ((attr (face-attribute 'default :background)))
               (if (or (null attr) (eq attr 'unspecified)) "#f9f5d7" attr))))
    (when (and (> width 0) (executable-find "magick") (file-exists-p src))
      (call-process "magick" nil nil nil src
                    "-background" bg "-flatten"
                    "-resize" (format "%dx" width)
                    dest)
      (setq fancy-splash-image dest)
      (run-with-timer 0.3 nil
        (lambda ()
          (when (fboundp '+doom-dashboard-reload)
            (+doom-dashboard-reload t)))))))

(add-hook 'after-make-frame-functions #'+my/scale-splash-image)
#+end_src
